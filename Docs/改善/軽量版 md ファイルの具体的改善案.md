# 軽量版 md ファイルの具体的改善案

## 結論

今の構成を完全に捨てる必要はありません。改善の要点は、**毎回読むファイルを極端に減らすこと**と、**常時ルールと参考資料を分離すること**です。Claude Code では、ルールが正しいこと以上に、**必要なときだけ必要な情報を読む構造**が重要です。

以下では、現行の思想をできるだけ残しつつ、**そのまま差し替えや再編に使える軽量版の具体案**を示します。

---

## 推奨ディレクトリ構成

| パス | 役割 | 常時読むか |
|---|---|---|
| `CLAUDE.md` | 入口。最小限の共通ルールだけ置く | はい |
| `.claude/workflow.md` | タスク規模別の進め方 | 必要時のみ |
| `.claude/rules.md` | 共通ルール | 必要時のみ |
| `.claude/pm-rules.md` | 要件・権限・仕様更新の判断軸 | 要件変更時 |
| `.claude/architect-rules.md` | 設計・責務分離・DB方針 | 設計変更時 |
| `.claude/coder-rules.md` | 実装原則・セキュリティ | 実装時 |
| `.claude/designer-rules.md` | UI変更時の最小デザインルール | UI変更時 |
| `.claude/tester-rules.md` | テストの必須原則 | テスト時 |
| `.claude/references/*` | サンプルコードや詳細仕様 | 原則読まない |

---

## まず変えるべき設計原則

### 原則1: `CLAUDE.md` は入口専用にする

`CLAUDE.md` に多くを書きすぎると、毎ターン無駄に文脈へ載りやすくなります。ここは**1ページで読める分量**に抑えるのが理想です。

### 原則2: `workflow.md` は全件強制にしない

現状の最大ボトルネックは、すべての依頼で多ロール・相互レビューを求める点です。品質管理としては立派ですが、AI運用では**毎回フル監査を要求している状態**に近く、トークン消費が大きくなります。

### 原則3: 役割ファイルは「rules」と「reference」に分ける

サンプルコードや詳細例は教育資料としては有効ですが、毎回読むルールファイルには重すぎます。**常設ルールは短く、例示は別紙**が基本です。

## 差し替え用サンプル: `CLAUDE.md`

以下は、そのままベースとして使える軽量版の例です。

```md
# CLAUDE.md

このリポジトリでは、まず本ファイルを読み、変更内容に応じて必要な補助資料だけ参照すること。

## プロジェクト概要
- アプリ名: こどちょ
- 概要: 子供の貯蓄情報を家族で管理するWebアプリ
- Backend: Spring Boot 3.x / Java 21 / PostgreSQL / Maven
- Frontend: Thymeleaf / Tailwind CSS / Chart.js
- Auth: Spring Security + Google OAuth2.0

## 常時守るルール
- `main` へ直接 push しない
- DBスキーマ変更時は Flyway マイグレーションを追加する
- `.env` はコミットしない
- Service 層にビジネスロジックを集約し、Controller は薄く保つ
- 権限チェックと所有者チェックを必ず行う
- UI変更時は `.claude/designer-rules.md` を参照する
- テスト追加・修正時は `.claude/tester-rules.md` を参照する

## 変更内容ごとの参照先
- 要件・権限・画面遷移: `.claude/pm-rules.md`
- 設計・責務分離・DB方針: `.claude/architect-rules.md`
- Java実装・Security: `.claude/coder-rules.md`
- UI/スタイル: `.claude/designer-rules.md`
- テスト方針: `.claude/tester-rules.md`
- 共通規約: `.claude/rules.md`
- 大規模変更の進め方: `.claude/workflow.md`

## 作業方針
- 軽微修正では必要な範囲だけ確認する
- 通常実装では、要件確認 → 設計 → 実装 → テスト → 必要なレビューの順で進める
- 大規模変更や権限変更では `.claude/workflow.md` を参照する
```

### この形の狙い

| 残したもの | 理由 |
|---|---|
| 技術スタック最小限 | 実装判断の前提になる |
| 絶対ルール | 毎回必要 |
| 参照先の地図 | 必要時だけ深掘りできる |

| 削ったもの | 理由 |
|---|---|
| ロール詳細一覧 | 毎回は不要 |
| 詳細レビュー観点 | 必要時参照で十分 |
| 長い説明文 | 入口ファイルとしては重い |

## 差し替え用サンプル: `.claude/workflow.md`

現状の思想を残しつつ、軽量化した案です。

```md
# 作業ワークフロー

このファイルは、通常より影響範囲が広い変更を行うときに参照する。

## 基本方針
- 変更内容に応じて必要なロールだけ使う
- 小規模変更は単一担当で完了してよい
- 横断的変更や権限変更では PM / Architect / Tester を含める
- UI変更時のみ Designer を必須とする
- レビューは全員一律ではなく、影響を受けるロールだけ実施する

## 作業区分
### 1. 軽微変更
対象: 文言修正、軽微なCSS調整、単純バグ修正、コメント修正

進め方:
1. 対象ファイルの確認
2. 最小修正
3. 必要ならテスト更新
4. 影響範囲の簡易確認

### 2. 通常変更
対象: 機能追加、バリデーション変更、Controller/Service修正、画面項目追加

進め方:
1. 要件確認
2. 設計確認
3. テスト観点整理
4. 実装
5. 必要なレビュー

### 3. 大規模変更
対象: 権限変更、DB設計変更、画面遷移変更、複数レイヤー横断の機能追加

進め方:
1. PM が要件と影響範囲を確認
2. Architect が設計方針を定義
3. Tester が主要テスト観点を整理
4. Coder / Designer が実装
5. 関係ロールがレビュー
6. 必要な仕様書を更新

## レビュー方針
- 要件変更がないなら PM レビューは省略可
- UI変更がないなら Designer レビューは不要
- テスト影響が小さい軽微修正では Tester は簡易確認でよい
- NG が出た場合は、指摘したロールだけ再確認する

## 仕様書更新
変更が発生した資料のみ更新する。
- 要件・権限変更: `Docs/仕様書/要件定義書.md`
- 画面項目・レイアウト変更: `Docs/仕様書/画面仕様書.md`
- ロジック変更: `Docs/仕様書/処理定義書.md`
- DB変更: `Docs/仕様書/構成図集.md`
- テスト変更: `Docs/仕様書/テスト仕様書.md`
```

### 改善ポイント

| 現状の問題 | 改善後 |
|---|---|
| 全件で多ロール強制 | 変更規模で分岐 |
| 相互レビュー前提 | 影響範囲ベースに縮小 |
| Windows絶対パスが多い | 相対パスに統一 |
| サブエージェント前提が強い | 必要時のみ複数観点へ |

## 差し替え用サンプル: `.claude/rules.md`

現行は比較的よくまとまっています。軽量化では、**運用ルールだけ残し、教材的な長文説明を削る**のが有効です。

```md
# 共通ルール

## ブランチ
- `main` へ直接 push しない
- `feature/<機能名>` `fix/<バグ名>` `test/<対象>` を使う

## コミット
- 形式: `<type>: <変更内容>`
- type: `feat` `fix` `refactor` `test` `docs` `style` `chore`
- summary は日本語で簡潔に書く
- 本文には「なぜ変更したか」を書く

## コメント
- コードから読める事実は書かない
- 背景、制約、意図、例外理由を書く

## 実装姿勢
- 変更は最小範囲に留める
- 無関係なリファクタを混ぜない
- 仕様変更を伴う場合のみ関連ドキュメントを更新する
```

### 補足

元の `rules.md` は教育資料としては良いです。ただし、Claude Code の常設ルールとしては、例文が多すぎると毎回の読み込み対象になりやすいため、**例文は reference 側へ退避**した方が効率的です。

## 差し替え用サンプル: `.claude/pm-rules.md`

```md
# PMルール

## 確認対象
- 画面一覧
- URL
- ロール権限
- ドメインルール
- 仕様書更新の要否

## ロール権限
- PARENT: 全操作
- PARTNER(EDIT): 入出金・目標の編集可
- PARTNER(VIEW): 閲覧のみ
- GRANDPARENT: 閲覧のみ
- CHILD: 自分の詳細のみ閲覧

## ドメインルール
- 残高は `入金合計 - 出金合計` で計算し、DBに保持しない
- 家族純資産は children の残高合計
- 目標達成は `残高 >= 目標金額` で判定し、達成フラグは持たない
- 学年ラベルは birth_date から動的に算出する
- PARENT は自分が登録した子供のデータのみ操作可能
- 子供削除時は関連データも削除する

## PMレビューの必須観点
- 要件逸脱がないか
- 権限が正しいか
- 画面遷移に破綻がないか
- 仕様書更新が必要なら反映したか
```

### 何を削ったか

| 削除対象 | 理由 |
|---|---|
| 詳細な画面一覧表 | 必要時は仕様書参照で十分 |
| 環境変数一覧 | 実装時に毎回は不要 |
| 長い説明 | 必須判断軸だけで足りる |

## 差し替え用サンプル: `.claude/architect-rules.md`

```md
# Architectルール

## 設計原則
- Controller は薄く保ち、ビジネスロジックは Service に集約する
- Repository は永続化責務に限定する
- 所有者チェックと権限制御を全アクセスポイントで考慮する
- N+1 クエリや不要な多重取得を避ける
- DB変更時は Flyway を追加する

## 構成方針
- パッケージ名は `com.example.kodoucho` を使う
- 主なレイヤーは `controller / service / repository / entity / dto / security`

## Architectレビューの必須観点
- レイヤー責務が守られているか
- 所有権検証が漏れていないか
- クエリ効率に問題がないか
- 実装と仕様書に大きな齟齬がないか
```

### 何を reference に逃がすべきか

- 依存ライブラリ一覧
- ディレクトリ構成の詳細図
- 推奨実装順序
- よくある質問

これらは毎回不要です。

## 差し替え用サンプル: `.claude/coder-rules.md`

```md
# Coderルール

## 基本
- パッケージ名は `com.example.kodoucho`
- クラス名は UpperCamelCase、メソッド名は lowerCamelCase
- インデントはスペース4つ
- `@Valid` と `BindingResult` を適切に使う

## 実装原則
- Service にビジネスロジックを集約する
- Controller に複雑な判定を書かない
- String 比較は `.equals()` を使う
- 所有者チェックを伴わない直接参照を避ける

## Security
- パスワードは BCrypt でハッシュ化する
- CSRF を無効化しない
- URLレベルのアクセス制御は `SecurityConfig` に集約する
- ロール別表示制御は `sec:authorize` を使う

## Coderレビューの必須観点
- 実装が規約に沿っているか
- セキュリティ上の抜け漏れがないか
- 所有権検証があるか
- 無駄に複雑化していないか
```

### 何を削るか

Chart.js の個別ルールは、UIと可視化の詳細なので通常の実装原則とは分けた方がよいです。`references/chart-reference.md` に移すのを推奨します。

```md
# Designerルール

## 最重要ルール
- 1px border による区切りを多用しない

- デザイン禁止事項に反していないか
- カラーパレット逸脱がないか
- ボタンや入力の形状ルールが守られているか
```

### ポイント

配色トークンをすべてここへ置く必要はありません。**主要色だけ残して、詳細トークンは別紙**で十分です。

## 差し替え用サンプル: `.claude/tester-rules.md`

```md
# Testerルール

## 基本方針
- Red → Green → Refactor を守る
- 実装前にテスト観点を整理する
- Service / Controller テストで不必要に重い起動をしない

## テスト方針
- Service は Unit Test を基本とする
- Controller は `@WebMvcTest` を基本とする
- 結合確認が必要な場合のみ `@SpringBootTest` を使う
- 正常系、異常系、権限チェックを意識する

## カバレッジ目標
- Service層 80%以上を目標とする

## 禁止事項
- Service / Controller テストで安易に `@SpringBootTest` を使わない
- テスト間で状態共有しない
- Controller にビジネスロジックを書いたままテストしない

## Testerレビューの必須観点
- 変更内容に対応するテストがあるか
- 正常系・異常系・権限チェックが必要十分か
- テストが重すぎないか
```

### reference に逃がすべき内容

- `@MockBean` の具体例
- `LoginUser` のセットアップ例
- サンプルテストコード全文
- 実行コマンドの長い説明

これらは `.claude/references/tester-recipes.md` に切り出すのが適切です。

## reference ファイルの置き方

軽量化では、消すのではなく**退避**が重要です。

### 例: `.claude/references/tester-recipes.md`

ここには以下を移します。

| 移動対象 | 元ファイル |
|---|---|
| `@WebMvcTest` の `@MockBean` 例 | `tester.md` |
| `LoginUser` のセットアップ例 | `tester.md` |
| Unit Test / Controller Test のテンプレート | `tester.md` |
| Chart.js の実装詳細 | `coder.md` |
| 依存ライブラリ表 | `architect.md` |
| 完全なカラーパレット | `designer.md` |

これにより、必要なときだけ深掘りでき、通常作業では読ませずに済みます。

## 導入順序

PL/PM視点で言うと、一気に全部差し替えるより、**効果が大きい順に段階導入**するのが安全です。

| 優先度 | 対応 | 期待効果 |
|---|---|---|
| 1 | `CLAUDE.md` を軽量化 | 毎回の入口コストを削減 |
| 2 | `workflow.md` を条件付き化 | 無駄な多ロール運用を抑制 |
| 3 | `tester.md` を rules / recipes に分割 | 長文サンプルの常設読込を回避 |
| 4 | `architect.md` を分割 | 詳細設計資料の常時読込を抑制 |
| 5 | 残り role ファイルを短文化 | 全体最適 |

## モデル使い分けとの相性

この軽量版は、モデル切替と非常に相性が良いです。

| フェーズ | モデル運用 | 読ませるファイル |
|---|---|---|
| 要件整理・レビュー | 高性能モデル | `CLAUDE.md` + 必要な rules |
| 実装 | 軽量モデル | `CLAUDE.md` + `coder-rules.md` |
| UI修正 | 軽量モデル | `CLAUDE.md` + `designer-rules.md` |
| テスト追加 | 軽量モデル | `CLAUDE.md` + `tester-rules.md` |
| 大規模変更 | 高性能モデル | `CLAUDE.md` + `workflow.md` + 必要 rules |

つまり、**軽量版mdにすると、モデル切替の効果がさらに出やすくなります**。

## 最終提言

今の `.md` は、開発ルールとしてはよくできています。ただし Claude Code の制限対策という観点では、**良い情報を持ちすぎていて、常時読み込みには重い**状態です。

> **改善の本質は「情報を減らすこと」ではなく、「毎回読む情報を減らすこと」です。**

そのため、最初にやるべきなのは次の3点です。

1. `CLAUDE.md` を入口専用の短いファイルへ縮める。  
2. `workflow.md` の全件多ロール強制をやめる。  
3. `tester.md` と `architect.md` を rules / reference に分割する。  

これだけでも、体感でかなり軽くなる可能性があります。

## 次にできること

必要であれば次のどちらかまで続けられます。

| 次の対応 | 内容 |
|---|---|
| 実ファイル差し替え版を作る | あなたの現在ファイル名に合わせて、完成版 md 一式をそのまま使える形で作成する |
| 最小変更版を作る | 現行ファイル構成を維持したまま、削る文・残す文を赤入れ形式で示す |

おすすめは前者です。**実運用でそのまま使える完成版**まで作ると、すぐ効果検証できます。
