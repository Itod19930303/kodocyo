# 作業ワークフロー

このファイルは変更規模に応じた作業フローを定義する。
ユーザーが規模を指定しない場合、**Claudeが依頼内容から判断して適用する**。

---

## 変更規模の判定基準

| 規模 | 対象例 |
|---|---|
| 軽微 | 文言修正・CSS微調整・単純バグ修正・コメント修正 |
| 通常 | 機能追加・バリデーション変更・Controller/Service修正・画面項目追加 |
| 大規模 | 権限変更・DB設計変更・画面遷移変更・複数レイヤー横断機能 |

---

## 軽微変更

1. 対象ファイルの確認
2. 最小修正
3. 影響範囲の簡易確認

ロール別作業・レビュー不要。

---

## 通常変更

1. 要件確認（権限・ドメインルールへの影響を確認）
2. 設計確認（レイヤー責務・DB変更の有無を確認）
3. 実装
4. 影響するロールのレビュー観点で確認

### レビュー省略ルール
- 要件変更がなければ PM レビュー省略可
- UI変更がなければ Designer レビュー省略可
- テスト影響が小さければ Tester は簡易確認でよい

---

## 大規模変更

1. **PM** — 要件・権限・画面遷移の影響確認
2. **Architect** — 設計方針・DB変更・処理フロー定義
3. **Tester** — テスト観点整理・テストを先に書く（Red）
4. **Coder** — 実装（Green）
5. **Designer** — UI実装
6. **全ロール** — 以下の観点でレビュー

### PM レビュー観点
- [ ] 要件・ロール権限が正しく実装されているか
- [ ] 画面遷移が仕様と一致しているか
- [ ] ドメインルール7項目が守られているか

### Architect レビュー観点
- [ ] Controller薄く・Serviceにロジック集約されているか
- [ ] 所有権検証が全アクセスポイントで実施されているか
- [ ] N+1クエリが発生していないか

### Coder レビュー観点
- [ ] セキュリティ脆弱性（IDOR・SQLインジェクション等）がないか
- [ ] String比較に `.equals()` を使っているか
- [ ] `findByIdAndParent()` を経由しているか

### Designer レビュー観点
- [ ] No-Line Rule・角丸・純黒禁止が守られているか
- [ ] カラーパレット外の色が使われていないか
- [ ] ボタンは `rounded-full`、入力は `rounded-xl` かつ border なしか

### Tester レビュー観点
- [ ] 変更に対応するテストがあるか
- [ ] 正常系・異常系・権限チェックが揃っているか
- [ ] Service層カバレッジ80%以上を維持しているか（`mvn verify`）

---

## 仕様書の更新ルール

変更が発生した仕様書のみ更新する。

| 変更内容 | 更新する仕様書 |
|---|---|
| 画面・URL・ロール権限変更 | `Docs/仕様書/要件定義書.md` |
| 画面レイアウト・入力項目変更 | `Docs/仕様書/画面仕様書.md` |
| Service/Repository ロジック変更 | `Docs/仕様書/処理定義書.md` |
| DBスキーマ変更 | `Docs/仕様書/構成図集.md` |
| テスト追加・変更 | `Docs/仕様書/テスト仕様書.md` |
