# 作業ワークフロー定義

このファイルはClaudeが作業依頼を受けたときに従うべきワークフローを定義します。
**すべての作業依頼でこのワークフローに従うこと。**

---

## 基本原則

作業依頼を受けたときは、品質を上げるために **ロール別サブエージェントが作業・レビューを行う**。
単一エージェントで実装を完結させることは禁止。

---

## ワークフロー全体像

```
作業依頼受信
    ↓
【Phase 1: 作業】各ロールサブエージェントが並行作業
    ↓
【Phase 2: レビュー】各ロールサブエージェントが相互レビュー
    ↓
全ロールOK？ → YES → 完了
             → NO  → 指摘事項を修正 → Phase 2 に戻る（ループ）
```

---

## Phase 1: 作業フェーズ

各ロールの担当範囲：

| ロール | 担当作業 | 参照ファイル |
|---|---|---|
| PM | 仕様確認・要件定義書更新・ドメインルール整合チェック | `.claude/commands/pm.md` |
| アーキテクト | 設計検討・処理定義書・構成図更新 | `.claude/commands/architect.md` |
| コーダー | Java実装・SecurityConfig・Repository/Service/Controller | `.claude/commands/coder.md` |
| デザイナー | Thymeleafテンプレート・Tailwind CSS実装 | `.claude/commands/designer.md` |
| テスター | テスト仕様書作成 + TDDサイクル（Red→Green）実装 | `.claude/commands/tester.md` |

### 実装の順序（依存関係がある場合）

1. **PM** が仕様を確定（要件定義書更新）
2. **アーキテクト** が設計を確定（処理定義書・DB設計更新）
3. **テスター** がテスト仕様書を作成し、テストを先に書く（Red）
4. **コーダー** がテストを通す実装を行う（Green）
5. **デザイナー** がUIを実装する
6. **テスター** がRefactorフェーズでテスト品質を整備

---

## Phase 2: レビューフェーズ

全ロールが実装結果をレビューし、OKかNGを判定する。

### 各ロールのレビュー観点

#### PM レビュー観点
- [ ] 仕様書（`Docs/仕様書/要件定義書.md`・`画面仕様書.md`）に記載された要件を満たしているか
- [ ] ロール権限（`.claude/commands/pm.md`）が正確に実装されているか
- [ ] ドメインルール7項目がすべて守られているか
- [ ] 画面遷移が仕様と一致しているか

#### アーキテクト レビュー観点
- [ ] レイヤー責務の分離が適切か（Controller薄く・Service集約）
- [ ] `findByIdAndParent()` など所有権検証が全アクセスポイントで実施されているか
- [ ] N+1クエリが発生していないか
- [ ] 処理定義書（`Docs/仕様書/処理定義書.md`）と実装が一致しているか

#### コーダー レビュー観点
- [ ] `.claude/commands/coder.md` の規約に準拠しているか
- [ ] `==` でのString比較がないか → `.equals()` を使用
- [ ] セキュリティ脆弱性（IDOR、SQLインジェクション等）がないか
- [ ] `findById()` を直接使う箇所がないか（必ず `findByIdAndParent()` を経由）

#### デザイナー レビュー観点
- [ ] DESIGN.mdのNo-Line Rule・角丸・純黒禁止が守られているか
- [ ] カラーパレット外の色が使われていないか
- [ ] ボタンはpill形状（`rounded-full`）か
- [ ] 入力フィールドは `rounded-xl` かつborderなしか

#### テスター レビュー観点
- [ ] テスト仕様書（`Docs/仕様書/テスト仕様書.md`）に今回の変更に対応するテストケースが記載されているか
- [ ] Service層のカバレッジが80%以上を維持しているか（`mvn verify`）
- [ ] 正常系・異常系・権限チェックがすべてテストされているか
- [ ] `@WebMvcTest` に必要な `@MockBean` が揃っているか

---

## レビュー判定ルール

- **全ロールOK** → 作業完了
- **1つでもNG** → 指摘事項を修正し、**NGを出したロールが再レビュー**（全ロール再レビュー不要）
- 修正後に新たな問題が発生した場合 → 関係するロールがレビュー

---

## 仕様書の更新ルール

作業で変更があった場合、以下の仕様書を必ず更新する：

| 変更内容 | 更新する仕様書 | 担当ロール |
|---|---|---|
| 画面・URL・ロール権限変更 | `Docs/仕様書/要件定義書.md` | PM |
| 画面レイアウト・入力項目変更 | `Docs/仕様書/画面仕様書.md` | PM/デザイナー |
| Service/Repository ロジック変更 | `Docs/仕様書/処理定義書.md` | アーキテクト |
| DBスキーマ変更 | `Docs/仕様書/構成図集.md` | アーキテクト |
| テスト追加・変更 | `Docs/仕様書/テスト仕様書.md` | テスター |

---

## 仕様書格納先

```
D:\developWorkSpace\kodocyoApp\Docs\仕様書\
├── 要件定義書.md       # 画面一覧・ロール権限・ドメインルール
├── 画面仕様書.md       # 各画面の入出力・レイアウト仕様
├── 処理定義書.md       # Service/Repository の処理ロジック仕様
├── 構成図集.md         # DB設計・クラス図・フロー図
└── テスト仕様書.md     # テストケース一覧・カバレッジ目標（テスター担当）
```

## サブエージェントへの指示テンプレート

Claudeがサブエージェントを立てるときは以下の情報を必ず含める：

```
あなたは【ロール名】として作業します。
参照ファイル: D:\developWorkSpace\kodocyoApp\.claude\commands\【role.md】

【作業内容の説明】

作業後に以下をレビューしてOK/NGを判定してください：
【そのロールのレビュー観点（workflow.mdより）】

NGの場合は修正点を具体的に指摘すること。
```
